## AUTONOMOUS AI AGENT TEAM WITH REAL-TIME TRANSPARENCY AND HUMAN OVERSIGHT

# SECTION 1: FUNDAMENTAL PARADIGM - THE LIVING AGENT WORKSPACE

## 1.1 CORE EXPERIENCE PHILOSOPHY

This application is NOT a traditional dashboard with pages and forms. This is a LIVING WORKSPACE where users watch, monitor, and collaborate with an autonomous team of AI agents working in real-time.

The moment a user enters their website URL, they are transported into a command center view where they observe their personal AI team spring into action. Every agent becomes visible, their work streams live, their communications with each other are transparent, and approval requests appear as interactive prompts requiring human decision.

Think of this as a glass-walled office where you can see every employee working at their desk. You can walk up to any desk, see exactly what they are doing, read their notes, review their research, and provide direction. The agents never hide their work. Everything is observable, interruptible, and collaborative.

## 1.2 THE SINGLE ENTRY POINT PHILOSOPHY

The user journey begins with one action: entering their website URL. This single input triggers the entire autonomous workflow. The user does not navigate between pages to initiate different tasks. Instead, they enter the URL, then watch as their AI team autonomously determines what needs to be done, assigns work, executes tasks, and presents findings - all while the user observes in real-time.

The agents work on AUTOPILOT but pause at critical decision points, waiting for human approval before proceeding. The user is never left wondering what is happening. The user is never required to figure out what to do next. The agents drive the workflow while humans maintain control.

## 1.3 TRANSPARENCY AS THE FUNDAMENTAL PRINCIPLE

Every single action taken by every single agent must be visible to the user. There are no background processes hidden from view. There are no black boxes. When an agent thinks, the user sees the thinking. When an agent researches, the user sees the research happening. When an agent writes, the user sees words appearing. When agents communicate, the user sees the messages.

This transparency is not optional. It is the core product experience. Users pay for the ability to watch AI experts work on their behalf, learn from their processes, and intervene when desired.

# SECTION 2: THE COMMAND CENTER INTERFACE

## 2.1 PRIMARY VIEW - THE AGENT WAR ROOM

Upon entering a website URL, the user transitions into the Agent War Room. This is the primary interface for the entire application experience.

### Visual Layout Concept

The screen transforms into a mission control aesthetic. The Orchestrator Agent occupies a prominent central or top position, clearly identified as the team leader coordinating all activities. Surrounding or below the Orchestrator, each specialist agent has their own dedicated workspace panel.

These panels are not static cards. They are LIVING WORKSPACES that stream real-time activity. Text flows, progress bars move, status indicators pulse, and work products materialize before the user's eyes.

### The Orchestrator's Command View

The Orchestrator Agent's panel shows the master plan - a visual workflow diagram displaying all tasks identified, their dependencies, current status, and assigned agents. Users see exactly how the Orchestrator has decomposed their goal into subtasks and which agents are responsible for each.

The Orchestrator's communication log streams in real-time, showing messages sent to other agents, responses received, decisions made, and coordination activities. Users witness the leadership happening.

### Individual Agent Workspaces

Each specialist agent's workspace panel contains their current task assignment clearly displayed at the top, a real-time activity stream showing exactly what they are doing right now, their working documents and intermediate outputs visible and scrollable, progress indicators for their current task, status badges showing states like Working, Waiting for Approval, Completed, or Blocked, and quick action buttons for user intervention.

## 2.2 AGENT WORKSPACE DEEP DIVE VIEW

Users can click on any agent's workspace panel to expand into a full-screen detailed view of that agent's work. This deep dive view shows the complete history of everything this agent has done during the current session.

### Contents of Deep Dive View

The detailed view includes every research query the agent has made, every data source accessed with timestamps, every piece of information gathered, the agent's reasoning and decision-making process documented in real-time logs, all intermediate work products and drafts, communications sent to and received from other agents, any errors encountered and how they were resolved, and all outputs produced.

Users can scroll through this history, search within it, and export it for their records. Nothing is hidden.

### Intervention Capabilities

From the deep dive view, users can send direct instructions to the agent, ask the agent to explain their reasoning, request the agent to redo specific work differently, pause the agent's work entirely, or provide additional context or requirements.

## 2.3 THE APPROVAL QUEUE INTERFACE

When any agent reaches a decision point requiring human approval, an approval request appears prominently in the interface. These are not dismissible notifications that can be ignored. They are blocking requests that pause the workflow until addressed.

### Approval Request Design

Each approval request contains clear identification of which agent is requesting approval, exactly what they are asking approval for, the work they have completed leading to this request, the options available to the user such as approve, reject, modify, or request more information, and the implications of each choice.

### Approval Queue Management

If multiple approval requests accumulate, they stack in a visible queue. Users can address them in any order, but the interface makes clear which requests are blocking other work from proceeding.

### Real-Time Waiting Visualization

When an agent is waiting for approval, their workspace panel clearly shows this waiting state. An animated indicator communicates that this agent is paused, awaiting human decision. The agent does not proceed until approval is granted.

# SECTION 3: THE AGENT TEAM STRUCTURE AND VISIBILITY

## 3.1 THE ORCHESTRATOR AGENT - THE VISIBLE LEADER

### Real-Time Leadership Display

The Orchestrator's workspace shows the master workflow in a visual format resembling a flowchart or kanban-style board. Users see every task identified, which agent is assigned, current status, dependencies between tasks, and overall progress toward the goal.

### Communication Hub

All inter-agent communications route through or are visible in the Orchestrator's panel. Users see messages like the Orchestrator assigning tasks to the Crawler Agent, the Crawler Agent reporting completion, the Orchestrator then activating the Keyword Agent with the crawl data, and so forth.

### Decision Logging

Every decision the Orchestrator makes is logged with reasoning. Users see entries explaining why certain tasks were prioritized, why specific agents were chosen, and how the workflow was adapted based on findings.

## 3.2 WEBSITE CRAWLER AGENT - THE SCOUT

### Live Crawling Visualization

When the Crawler Agent works, users see pages being discovered in real-time. A visual representation shows the website structure being mapped, with nodes appearing as pages are found. URLs scroll in a live feed as they are accessed.

### Current Activity Display

The workspace shows exactly which page is being crawled right now, what information is being extracted, any issues encountered such as redirects or errors, and progress toward complete site coverage.

### Findings Stream

As the crawler discovers information, findings stream into view showing total pages found with count updating live, page types identified, content themes detected, technical issues discovered, and tone and voice analysis progressing.

### Approval Gate

Upon completing the crawl, the agent presents findings and requests approval to proceed. The user reviews discovered pages, can exclude sections, and approves the data for use by subsequent agents.

Users watch this agent analyze the crawled content, identify topics, expand into keyword territory, and query external APIs for volume and difficulty data.

### Data Source Transparency

Every API call is logged visibly. Users see which external services are being queried, what data is being retrieved, and response times.

The agent presents the complete keyword list for approval. Users can add, remove, reprioritize, or request additional research before approving the list for SERP analysis.

## 3.4 SERP ANALYSIS AGENT - THE RESEARCHER

### Parallel Analysis Visualization

When analyzing multiple keywords simultaneously, each keyword has its own visible progress stream. Users see parallel work happening across multiple keywords with independent progress tracking.

### Approval Gate

After analysis completes, the agent presents comprehensive findings. Users review competitor insights, strategic recommendations, and opportunity assessments before approving for content strategy development.

# SECTION 5: THE AUTOPILOT WITH GUARDRAILS MODEL

## 5.1 AUTONOMOUS OPERATION

After the user enters the URL and initiates the workflow, agents operate autonomously. They do not require constant user input to proceed. They make decisions, execute work, and drive the process forward.

The user can step away and return to find progress has been made. The agents work continuously until they reach an approval gate.

## 5.2 APPROVAL GATE SYSTEM

### Mandatory Approval Points

Certain decisions require human approval and cannot be bypassed. These include finalizing the keyword list for SERP analysis, approving the content strategy before article generation, approving internal link suggestions before embedding, approving article content before final export, and approving any significant strategic recommendations.

### Configurable Approval Points

Users can configure additional approval points or reduce approval requirements based on their trust level and desired involvement. Power users may want fewer interruptions. New users may want more checkpoints.

### Approval Timeout Handling

If approval is not provided within a configurable timeframe, agents can send reminder notifications, escalate to alternative contacts if configured, pause the entire workflow with clear status indication, or proceed with default choices if user has pre-authorized defaults.

## 5.3 USER INTERVENTION AT ANY POINT

Even when not at an approval gate, users can intervene at any time. They can pause any agent immediately, redirect an agent's focus, provide additional instructions, override an agent's approach, or request explanation of current activity.

This intervention capability does not disrupt the autopilot model. It enhances it by giving users confidence that they maintain control even while agents operate autonomously.

# SECTION 6: REAL-TIME PROGRESS AND STATUS SYSTEM

## 6.1 GLOBAL PROGRESS OVERVIEW

A persistent overview element shows the overall workflow progress. This displays what percentage of the total workflow is complete, which phase is currently active such as crawling, researching, analyzing, strategizing, or writing, how many agents are currently active, how many approval requests are pending, and estimated time to next milestone.

Each agent's workspace panel displays clear status through visual states.

Every agent maintains a real-time log of their activity. This log streams in the agent's workspace panel and is preserved for review. Log entries include timestamps with precise timing, action descriptions of what the agent did, results achieved from each action, decisions made with their reasoning, and data points gathered or produced.

## 6.4 PROGRESS METRICS

Quantifiable progress metrics update in real-time. These include pages crawled out of estimated total, keywords discovered with volume indicating completeness, keywords analyzed out of total queued, articles in progress with word count out of target, and images generated out of required.

# SECTION 7: THE UNIQUE INTERFACE DESIGN REQUIREMENTS

## 7.1 BREAKING FROM CONVENTION

This interface must NOT look like a standard SaaS dashboard. It must NOT look like a typical analytics tool. It must NOT look like a content management system.

This interface should feel like Mission Control, a War Room, a command center where serious work is being coordinated by an elite team. Yet it must also feel welcoming, elegant, and luxuriously minimal rather than overwhelming or intimidating.

## 7.2 THE NOTION-INSPIRED FOUNDATION

Take Notion's minimalist elegance, generous whitespace, and typographic clarity as the baseline. Then evolve it into something that supports real-time activity visualization while maintaining that clean, premium aesthetic.

### Clean Typography

Use typefaces that are highly readable for streaming text content. Body text must remain legible even when scrolling rapidly.

### Generous Whitespace

Despite the density of information, maintain breathing room. Let panels have margins. Let content have padding. Never let the interface feel cramped.

### Subtle Color Application

Reserve color for meaning. Status indicators use color purposefully. Accent colors highlight important elements. The base remains neutral and elegant.

## 7.3 THE MASCOT INTEGRATION

A distinctive mascot character serves as the friendly face of the platform. This mascot appears strategically throughout the experience providing welcome and onboarding guidance, celebration when milestones are achieved, helpful tips during waiting periods, personality during loading states, and encouragement when users seem stuck.

The mascot must feel premium and clever, not childish. Think sophisticated character design that appeals to professionals.

## 7.4 THE THEME TOGGLE

The theme toggle mechanism must be unique and memorable. Consider creative implementations that surprise and delight.

### Comprehensive Theme Application

Every element transforms between light and dark themes including all agent workspace panels, all text and backgrounds, all charts and visualizations, all status indicators, all interactive elements, and all mascot appearances.

### Contrast Compliance

Both themes must meet accessibility contrast requirements. All text must be readable in both modes.

### Luxurious Transitions

Theme switching triggers smooth, satisfying transitions. Colors interpolate gracefully. Elements fade and transform elegantly. The transition itself is a small moment of delight.

## 7.5 STREAMING TEXT AESTHETICS

Since much of the interface involves text streaming in real-time, the aesthetics of streaming text require careful consideration. Character-by-character streaming for dramatic effect where appropriate, word-by-word streaming for readability during article generation, smooth scroll behavior as content grows, subtle cursor or insertion point indication, and appropriate animation speeds that feel natural all require attention.

## 7.6 RESPONSIVE DESIGN FOR MONITORING

The interface must work on all screen sizes, but the experience may legitimately differ between devices.

On desktop, users get the full war room experience with all agents visible simultaneously.

On tablet, users get a condensed view with easy switching between agent panels.

On mobile, users get a focused single-agent view with quick navigation between agents.

All versions maintain the real-time streaming and monitoring capabilities.

## 7.7 COLLAPSIBLE ELEMENTS

The sidebar for navigation must be collapsible, maximizing workspace for the agent war room. Agent panels should be expandable and collapsible. The communication stream can be toggled visible or hidden. Approval queue can expand for attention or collapse when empty.

# SECTION 12: DETAILED AGENT WORKFLOW SPECIFICATIONS

## 12.1 COMPLETE WORKFLOW SEQUENCE

### Phase 1: Initiation

User enters website URL. The Orchestrator Agent activates, displaying its workspace. The Orchestrator announces the mission and presents an initial workflow plan. User may approve plan or request modifications.

### Phase 2: Website Understanding

The Orchestrator assigns the Crawler Agent. The Crawler Agent's workspace activates showing live crawling progress. Pages are discovered and displayed in real-time. Content is analyzed and tone is assessed. Upon completion, the Crawler presents findings for approval. User reviews and approves or requests modifications. Approved data flows to the next phase.

### Phase 3: Keyword Discovery

The Orchestrator assigns the Keyword Research Agent with crawl data. The Keyword Agent's workspace shows research activity. Keywords are discovered and displayed with live API data population. Keywords are grouped into thematic clusters. Upon completion, the keyword list is presented for approval. User reviews, modifies if needed, and approves. Approved keywords flow to SERP analysis.

### Phase 4: SERP Analysis

The Orchestrator assigns the SERP Analysis Agent with approved keywords. Users can select which keywords to analyze or analyze all. The SERP Agent's workspace shows per-keyword analysis progress. Parallel analysis of multiple keywords is visible. People Also Ask questions appear as discovered, with minimum seven to ten per keyword. AI Overview analysis is captured and displayed. Competitor analysis with domain authority, traffic, and ranking changes populates. Upon completion, comprehensive SERP report is presented for approval. User reviews strategic insights and approves for strategy development.

### Phase 5: Strategy Development

The Orchestrator assigns the Content Strategy Agent with all gathered intelligence. The Strategy Agent's workspace shows strategic thinking process. Topic clusters are built visibly. Content calendar recommendations develop. Internal linking strategy forms. The complete strategy is presented for approval. User reviews, modifies, and approves. Approved strategy authorizes content creation.

### Phase 6: Content Creation

The Orchestrator assigns the Article Generation Agent with strategy and supporting data. User selects which articles to generate from the strategy. Parallel article generation activates for selected articles. Each article has its own visible workspace panel. Pre-writing research is visible for each article. Outlines develop and can be reviewed before writing proceeds. Live writing streams words in real-time. Image generation integrates at appropriate points. Internal link suggestions are presented for approval. External links are researched and added transparently. AI Overview optimization pass is visible. People Also Ask answers are explicitly incorporated. Upon each article completion, immediate download is available. The complete batch completes with all articles downloadable.

### Phase 7: Ongoing Monitoring

The Rank Tracking Agent begins monitoring published content. The Analytics Agent connects to user's Google Analytics if authorized. The Optimization Agent monitors for improvement opportunities. These agents work continuously with their activity visible in the war room.

## 12.2 CONDITIONAL WORKFLOW BRANCHES

The workflow adapts based on findings and user decisions. If the crawl reveals limited content, additional external research may be recommended. If keyword research reveals high-competition topics, strategy adjustments are proposed. If SERP analysis shows established competitors, differentiation strategies are emphasized. If Analytics data shows underperforming content, optimization becomes priority.

The Orchestrator manages these adaptive branches with full transparency.

# SECTION 13: ARTICLE GENERATION DEEP SPECIFICATIONS

## 13.1 MANDATORY REQUIREMENTS ENFORCEMENT

Every generated article must exceed three thousand words. This is enforced at the agent level with visible word count targeting. Articles that would fall short trigger additional content development automatically.

Every article must be structurally unique. The agent must not follow templates. Each article develops its own organization based on the topic requirements.

Every article must include original images. The featured image is mandatory. Content images numbering four to six are mandatory. Images are generated and embedded during the writing process.

Every article must include data visualizations where appropriate. Charts, tables, and graphs enhance content with interactive features and citations.

Every article must answer all relevant People Also Ask questions comprehensively.

Every article must be optimized for AI Overview inclusion following the SERP analysis insights.

Every article must include internal links to the user's existing content based on sitemap analysis.

Every article must include external links to high-authority relevant sources. Link quantity is not artificially limited. Links are included as needed for the article to be maximally valuable.

Every link must be validated as live and working before article finalization. Broken links are absolutely prohibited.

## 13.2 THE AI OVERVIEW OPTIMIZATION SYSTEM PROMPT

The AI Overview Optimizer component uses the following system prompt guidance:

You are an expert content writer for the user's niche website. Your goal is to write a comprehensive, high-quality, and trustworthy article about the target keyword. The article should be optimized for Google's AI Overviews and demonstrate strong E-E-A-T signals.

Target Query: The specific long-tail user question being addressed.

Instructions for Audience: Address the described target audience appropriately.

Instructions for Tone: Maintain the specified tone such as confident, helpful, and conversational, but professional.

Instructions for Structure: Start with a concise, direct answer in two to three sentences to the Target Query. Break down the main topic into logical sections using descriptive H2 and H3 headings. Phrase headings as questions where appropriate. Use numbered lists for step-by-step instructions and bullet points for lists of items or benefits. Keep paragraphs short with two to three sentences maximum.

Instructions for E-E-A-T and Credibility: Incorporate real-world examples, practical tips, or unique insights from first-hand experience. Include statistics or data points to back up claims, and naturally link to authoritative external sources such as university extension sites and research institutions. Mention the author's credentials or expertise in the content or suggest an author bio with qualifications.

Instructions for Optimization: Ensure the main topic and related entities and synonyms are used naturally throughout the text for semantic depth. Avoid jargon unless necessary, and if used, provide a brief definition. Ensure all information is accurate and up-to-date.

Format: The final output should be a complete article draft, ready for human review and refinement. Do not include filler phrases or disclaimers.

## 13.3 UNIQUENESS REQUIREMENTS

Each article must have its own unique research pathway exploring different sources and angles. Each article must have its own unique structural approach to organizing information. Each article must have its own unique image concepts and placements. Each article must have its own unique data visualization choices. Each article must have its own unique voice within the brand tone parameters.

The system explicitly avoids templated responses. Parallel articles must diverge in their approaches while maintaining quality standards.

## 13.4 HUMAN-LIKE CONTENT REQUIREMENTS

Articles must feel human-written. This means natural language flow without robotic patterns, varied sentence structures with differing lengths and complexity, authentic opinions and perspectives where appropriate, emotional resonance where relevant to the topic, and imperfect but authentic phrasing rather than overly polished machine text.

The content should pass any reasonable human detection assessment because it genuinely embodies human thinking and communication patterns.

// ─── Helper: Exponential Backoff Retry ───────────────────────────────────────
const withRetry = async <T>(
  operation: () => Promise<T>,
  retries = 3,
  baseDelay = 2000
): Promise<T> => {
  try {
    return await operation();
  } catch (error: any) {
    const isRateLimit =
      error.status === 429 ||
      error.code === 429 ||
      error.message?.includes("429") ||
      error.message?.includes("quota");
    const isServerOverload = error.status === 503 || error.code === 503;

    if (retries > 0 && (isRateLimit || isServerOverload)) {
      const delay = baseDelay * Math.pow(2, 3 - retries);
      console.warn(
        `API Rate Limit hit. Retrying in ${delay}ms... (${retries} attempts left)`
      );
      await new Promise((resolve) => setTimeout(resolve, delay));
      return withRetry(operation, retries - 1, baseDelay);
    }
    throw error;
  }
};

    const jsonStr = jsonMatch ? jsonMatch[1] : text;
    return JSON.parse(jsonStr);

    return null;
  }
};

export const applyLinkSuggestions = async (
  content: string,
  suggestions: LinkSuggestion[]
): Promise<string> => {
  let newContent = content;

  const sortedSuggestions = [...suggestions].sort(
    (a, b) => b.anchorText.length - a.anchorText.length
  );

  for (const suggestion of sortedSuggestions) {
    if (!suggestion.anchorText || !suggestion.targetUrl) continue;

    const escapedAnchor = suggestion.anchorText.replace(
      /[.*+?^${}()|[\]\\]/g,
      "\\$&"
    );

    // Only replace if NOT already inside a markdown link
    const regex = new RegExp(
      `(?<!\\[)(?<![\\(])${escapedAnchor}(?!\\])(?![\\)])`,
      "i"
    );

    newContent = newContent.replace(
      regex,
      `[${suggestion.anchorText}](${suggestion.targetUrl})`
    );
  }

  return Promise.resolve(newContent);
};

export const generateImage = async (
  prompt: string,
  aspectRatio: string = "16:9"
): Promise<string> => {
  try {
    const response: GenerateContentResponse = await withRetry(() =>
      ai.models.generateContent({
        model: "gemini-2.5-flash-image",
        contents: {
          parts: [{ text: prompt }],
        },
        config: {
          imageConfig: {
            aspectRatio: aspectRatio as any,
          },
        },
      })
    );

    for (const part of response.candidates?.[0]?.content?.parts || []) {
      if (part.inlineData) {
        return `data:image/png;base64,${part.inlineData.data}`;
      }
    }
    return `https://picsum.photos/seed/${encodeURIComponent(prompt)}/800/600`;
  } catch (e) {
    console.error("Image Gen Error", e);
    return `https://picsum.photos/seed/${encodeURIComponent(prompt)}/800/600`;
  }
};

export const checkRank = async (
  domain: string,
  keywords: string[]
): Promise<BulkRankResponse> => {
  const keywordList = keywords.join(", ");

  const prompt = `
<SYSTEM>

</SYSTEM>

<MISSION>
Check the current Google ranking positions for domain "${domain}" across the following keywords: ${keywordList}
</MISSION>

<INSTRUCTIONS>
For EACH keyword in the list, perform the following:

### STEP 1: SEARCH EXECUTION
- Execute a Google Search for the exact keyword.
- Scan organic results (positions 1-100).

### STEP 2: DOMAIN DETECTION
- Look for ANY page from "${domain}" in the results.
- Record the FIRST occurrence (highest ranking position).

### STEP 3: RESULT RECORDING
For each keyword, report:
- "keyword": The exact keyword searched.
- "rank": The numerical position (1-100). If domain is NOT found in visible results, set to 0.
- "url": The full URL of the ranking page from ${domain}. Null if not found.
- "title": The page title as shown in search results. Null if not found.
- "found": Boolean — true if domain was found, false otherwise.
- "analysis": A brief analysis (max 20 words) noting:
  * If found: relative position strength, competitor context, any featured snippets.
  * If not found: what types of domains ARE ranking (big brands? niche sites? forums?).

### STEP 4: EXECUTIVE SUMMARY
After checking ALL keywords, write a 2-3 sentence executive summary covering:
- Overall ranking performance (how many keywords ranking, average position).
- Strongest performing keyword(s).
- Biggest opportunities (keywords where domain is close to page 1).
- Critical gaps (important keywords where domain is not ranking at all).
</INSTRUCTIONS>

<OUTPUT_FORMAT>
Return a JSON object with the following structure. Use structured schema for reliable parsing.
</OUTPUT_FORMAT>

<RULES>
- You MUST use Google Search for every keyword check — no estimation.
- Rank must be an integer. 0 = not found in results.
- URL and title are null (not empty strings) when domain is not found.
- Analysis must be specific to what you ACTUALLY see in search results.
- The summary must be data-driven, referencing specific keyword performance.
- Do NOT conflate paid/ad results with organic rankings.
- Check the ENTIRE first page minimum (10 results) for each keyword.
</RULES>
`;

  const schema: Schema = {
    type: Type.OBJECT,
    properties: {
      results: {
        type: Type.ARRAY,
        items: {
          type: Type.OBJECT,
          properties: {
            keyword: { type: Type.STRING },
            rank: {
              type: Type.INTEGER,
              description: "The numerical rank position. 0 if not found.",
            },
            url: {
              type: Type.STRING,
              description: "The full URL of the ranking page.",
            },
            title: {
              type: Type.STRING,
              description: "The title of the page.",
            },
            analysis: {
              type: Type.STRING,
              description: "Brief analysis of the result.",
            },
            found: {
              type: Type.BOOLEAN,
              description: "True if domain was found in results.",
            },
          },
          required: ["keyword", "rank", "analysis", "found"],
        },
      },
      summary: {
        type: Type.STRING,
        description: "Overall summary of performance.",
      },
    },
    required: ["results", "summary"],
  };

  try {
    const response: GenerateContentResponse = await withRetry(() =>
      ai.models.generateContent({
        model: "gemini-3-pro-preview",
        contents: prompt,
        config: {
          tools: [{ googleSearch: {} }],
          responseMimeType: "application/json",
          responseSchema: schema,
        },
      })
    );

    const result = JSON.parse(response.text || "{}");

    const groundingChunks =
      (response.candidates?.[0]?.groundingMetadata
        ?.groundingChunks as GroundingChunk[]) || [];
    const groundingUrls = groundingChunks
      .map((chunk) => chunk.web?.uri)
      .filter((uri): uri is string => !!uri);

    return {
      results: result.results || [],
      summary: result.summary || "No data returned.",
      groundingUrls,
    };
  } catch (error) {
    console.error("Rank Tracking Error:", error);
    throw new Error("Failed to retrieve ranking data. Please try again.");
  }
};

export const predictIndexingSuccess = async (
  content: string,
  keyword: string,
  url: string,
  siteAuthority: string
): Promise<{
  likelihood: string;
  estimatedTimeToIndex: string;
  technicalChecklist: {
    factor: string;
    status: string;
    action: string;
  }[];
  indexingStrategy: string;
  submissionAdvice: string;
}> => {
  const safeContent = sanitizeContentForContext(content);

  const prompt = `
<SYSTEM>
You are the INDEXING AGENT — the technical SEO specialist of the war room. You analyze content and technical factors to predict how quickly and successfully Google will index new content.

You understand Google's crawl and indexing pipeline:
- Crawl budget allocation based on site authority.
- Content quality thresholds for indexing.
- Technical factors that accelerate or block indexing.
- Sitemap and internal linking effects on discovery speed.
- Google Search Console submission mechanics.

Your predictions and recommendations help the operator take immediate action to accelerate indexing after content publication.
</SYSTEM>

<MISSION>
Predict indexing success for new content:
- URL: ${url}
- Target Keyword: "${keyword}"
- Site Authority Level: ${siteAuthority}
</MISSION>

<CONTENT>
${safeContent.substring(0, 3000)}
</CONTENT>

<INSTRUCTIONS>

### PREDICTION 1: INDEXING LIKELIHOOD
Assess the overall likelihood this content will be indexed: "High" | "Medium" | "Low"
Base this on: content quality/depth, keyword targeting, site authority, and technical factors.

### PREDICTION 2: TIME TO INDEX
Estimate how long until Google indexes this page:
- High authority sites: "1-3 days"
- Medium authority: "3-7 days"
- Low authority: "1-4 weeks"
- Consider content quality as an accelerator/decelerator.

### PREDICTION 3: TECHNICAL CHECKLIST
Evaluate these technical indexing factors:
1. Content Length — Is it substantial enough? (>1000 words = good)
2. Keyword Optimization — Is the target keyword present in title, H1, first paragraph, and throughout?
3. Internal Links — Does the page receive internal links from existing pages?
4. Meta Tags — Are title tag and meta description optimized?
5. Schema Markup — Is structured data present/recommended?
6. Canonical URL — Is the canonical properly set?
7. Mobile Friendliness — Any mobile concerns?
8. Page Speed — Any speed concerns based on content complexity?

For each factor: "Pass" | "Warning" | "Fail", with specific action if not Pass.

### PREDICTION 4: INDEXING STRATEGY
Write a 2-3 sentence strategy for getting this content indexed as quickly as possible.

### PREDICTION 5: SUBMISSION ADVICE
Provide specific Google Search Console submission advice:
- Should they submit via URL Inspection tool?
- Should they update the sitemap?
- Any additional technical steps?
</INSTRUCTIONS>

<OUTPUT_FORMAT>
Return JSON:
{
  "likelihood": "High|Medium|Low",
  "estimatedTimeToIndex": "string",
  "technicalChecklist": [
    {
      "factor": "string",
      "status": "Pass|Warning|Fail",
      "action": "string — specific action if not Pass, 'None needed' if Pass"
    }
  ],
  "indexingStrategy": "string",
  "submissionAdvice": "string"
}
</OUTPUT_FORMAT>

<RULES>
- Checklist MUST include all 8 technical factors listed.
- Likelihood assessment must correlate with checklist results — if 3+ factors Fail, likelihood cannot be High.
- Time estimates must correlate with stated site authority level.
- Actions must be SPECIFIC enough for a developer/SEO to implement immediately.
- Do NOT recommend actions that require paid tools — only free/accessible methods.
- Indexing strategy must include at least one action beyond "submit to Google Search Console."
</RULES>
`;

  try {
    const response: GenerateContentResponse = await withRetry(() =>
      ai.models.generateContent({
        model: "gemini-3-flash-preview",
        contents: prompt,
        config: { responseMimeType: "application/json" },
      })
    );
    const data = cleanAndParseJSON(response.text || "") || {};
    return {
      likelihood: data.likelihood || "Medium",
      estimatedTimeToIndex: data.estimatedTimeToIndex || "3-7 days",
      technicalChecklist: Array.isArray(data.technicalChecklist)
        ? data.technicalChecklist
        : [],
      indexingStrategy:
        data.indexingStrategy || "Submit via Google Search Console.",
      submissionAdvice:
        data.submissionAdvice ||
        "Use URL Inspection tool in GSC to request indexing.",

  } catch (e) {
    return {
      likelihood: "Medium",
      estimatedTimeToIndex: "Unknown",
      technicalChecklist: [],
      indexingStrategy: "Submit to GSC immediately.",
      submissionAdvice: "Use URL Inspection tool to request indexing.",

export const validateArticleLinks = async (
  content: string
): Promise<{
  totalLinks: number;
  validLinks: { url: string; anchor: string; type: string }[];
  brokenLinks: { url: string; anchor: string; issue: string; replacement: string }[];
  summary: string;
}> => {
  const safeContent = sanitizeContentForContext(content);

  const prompt = `
<SYSTEM>
You are the LINK VALIDATION AGENT — the quality assurance specialist. Before any article is finalized, you verify that EVERY link in the content is live, working, and pointing to the correct destination.

Broken links are ABSOLUTELY PROHIBITED in final output. Your job is to catch every broken link before the operator downloads the article.

Use Google Search to verify that linked pages exist and are accessible.
</SYSTEM>

<MISSION>
Validate every link in the following article content.
</MISSION>

<CONTENT>
${safeContent.substring(0, 10000)}
</CONTENT>

<INSTRUCTIONS>
1. Extract EVERY Markdown link from the content: [anchor text](URL)
2. For EACH link:
   - Classify as "internal" or "external" based on URL.
   - Use Google Search to verify the URL exists and is accessible.
   - If the link appears broken, find a REPLACEMENT URL to the same or similar content.
3. Report total links found.
4. Separate valid and broken links.
5. For broken links: provide the original URL, what's wrong with it, and a suggested replacement.
6. Write a summary.
</INSTRUCTIONS>

<OUTPUT_FORMAT>
Return JSON:
{
  "totalLinks": number,
  "validLinks": [
    { "url": "string", "anchor": "string", "type": "internal|external" }
  ],
  "brokenLinks": [
    { "url": "string", "anchor": "string", "issue": "string", "replacement": "string" }
  ],
  "summary": "string"
}
</OUTPUT_FORMAT>

<RULES>
- EVERY link in the content must be accounted for — either in validLinks or brokenLinks.
- Broken links MUST have a replacement suggestion.
- Use Google Search to verify — do not just check URL format.
- Image links (data:image/...) should be EXCLUDED from validation.
- Replacement URLs must be to real, accessible pages.
</RULES>
`;

  try {
    const response: GenerateContentResponse = await withRetry(() =>
      ai.models.generateContent({
        model: "gemini-3-flash-preview",
        contents: prompt,
        config: {
          responseMimeType: "application/json",
          tools: [{ googleSearch: {} }],
        },
      })
    );
    const data = cleanAndParseJSON(response.text || "") || {};
    return {
      totalLinks: data.totalLinks || 0,
      validLinks: Array.isArray(data.validLinks) ? data.validLinks : [],
      brokenLinks: Array.isArray(data.brokenLinks) ? data.brokenLinks : [],
      summary: data.summary || "Validation completed.",
    };
  } catch (e) {
    console.error("Link Validation Failed", e);
    return {
      totalLinks: 0,
      validLinks: [],
      brokenLinks: [],
      summary: "Link validation failed — manual review recommended.",
    };
  }
};
